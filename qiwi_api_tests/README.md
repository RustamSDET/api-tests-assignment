# QIWI Payout API Tests

Проект содержит автотесты для QIWI Payout API на базе Pytest, Playwright (в качестве HTTP-клиента) и Pydantic v2, а также коллекцию запросов Postman.

## Структура проекта

- **qiwi_api_tests/**
  - **requirements.txt** — Список необходимых зависимостей (pytest, playwright, pydantic).
  - **pytest.ini** — Базовые настройки запуска тестов.
  - **schemas.py** — Модели валидации ответов сервера на Pydantic v2.
  - **api_client.py** — HTTP-клиент для работы с QIWI API.
  - **conftest.py** — Сессионные фикстуры для инициализации Playwright и клиента.
  - **tests/**
    - **test_payout.py** — Интеграционные сценарии тестов.

---

## Описание автотестов в Python (pytest)

В файле `tests/test_payout.py` реализовано 3 основных тест-кейса:

1. **`test_service_availability`** (Доступность сервиса)  
   Запрашивает список платежей (GET-запрос) и проверяет, что сервер возвращает успешный код ответа `200`, а структура ответа валидируется схемой `PaymentInfo`.

2. **`test_get_balance`** (Запрос баланса)  
   Запрашивает баланс точки (GET-запрос), проверяет статус `200`, валидирует ответ через схему `BalanceInfo` и контролирует ключевое условие: числовое значение баланса (`balance.float_value`) должно быть строго больше `0.0`.

3. **`test_payout_lifecycle`** (Жизненный цикл платежа: Создание и Исполнение)
   - **Шаг создания**: Генерирует уникальный UUID для платежа, отправляет PUT-запрос на создание выплаты в 1.00 RUB и проверяет, что статус платежа перешел в `"READY"`.
   - **Шаг исполнения**: Отправляет POST-запрос на исполнение созданного платежа и проверяет, что статус перешел в `"IN_PROGRESS"` или `"COMPLETED"`.

## Коллекция Postman (`test_assignment.postman_collection.json`)

В корне проекта находится файл `test_assignment.postman_collection.json` — это готовая коллекция запросов Postman для тестирования API.

### 1. Переменные коллекции (Collection Variables)

- `base_url` — базовый хост API (например, `api.qiwi.com`).
- `token` — авторизационный Bearer-токен.
- `paymentId` — уникальный идентификатор платежа (генерируется динамически).

### 2. Скрипты автоматизации (Pre-request и Tests)

- **Pre-request скрипты**:
  В запросе создания платежа используется JS-скрипт, генерирующий новый UUID (`uuid.v4()`) и автоматически сохраняющий его в переменную `paymentId` для подстановки в URL текущего и последующих запросов.
- **Скрипты тестирования (Tests / post-response)**:
  - В запросе списка платежей проверяется код ответа (`200 OK`).
  - В запросе баланса проверяется корректность данных — скрипт парсит JSON и утверждает, что баланс точки строго больше 0 рублей (`parseFloat(jsonData.balance.value) > 0`).
